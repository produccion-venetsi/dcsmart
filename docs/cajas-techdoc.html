Módulo de Cajas

Versión 2026-06-12 Branch feat/app-local-isolation Stack Fastify 4 · Prisma 5 · React 19

El módulo de Cajas registra los turnos operativos de cada local gastronómico. Cada caja agrupa tres tipos de información: los datos generales del turno (totales, cajero, fechas), los movimientos de caja (ingresos y egresos de dinero con método de pago) y los detalles (desglose por canal o concepto contable, sin atarse a un único método de pago). El acceso está estrictamente aislado por grupo (App) y local.

Qué es una Caja

Una Caja representa un turno de apertura y cierre de operaciones en un local. Es la unidad de registro diario: cada turno tiene un número identificador (nro_turno), un responsable (cajero), fechas de inicio y cierre, y los totales monetarios consolidados del turno.

Los totales del encabezado (total, efectivo, fiscal) son valores que el operador ingresa manualmente al cerrar el turno. No se calculan automáticamente a partir de los movimientos — los movimientos y detalles son registros complementarios de trazabilidad, no la fuente de verdad del total.

Origen externo: el campo origin (DCSMART / TAPTAP / FFUDO) indica si la caja fue creada desde este sistema o importada desde otro software de punto de venta. Las cajas de origen externo pueden llegar por integración y tienen los mismos campos que las nativas.

Aislamiento multi-tenant

Todas las rutas del módulo de cajas están protegidas por el middleware appContext. Este middleware requiere que cada request incluya el header X-App-Id con el UUID del grupo activo. A partir de ese header, resuelve qué locales puede ver el usuario (allowedLocalIds) y rechaza con 403 si intenta acceder a datos de otro grupo.

Cómo se resuelven los locales permitidos

// plugins/appContext.js
if (rol === 'super_admin') {
  // Todos los locales activos del grupo seleccionado
  request.allowedLocalIds = locales.map(l => l.id)
} else {
  const accesos = await findMany({ id_user, id_app })
  if (accesos.length > 0) {
    // Solo los locales asignados explícitamente (UserLocalAccess)
    request.allowedLocalIds = accesos.map(a => a.id_local)
  } else {
    // Sin restricción → todos los locales del grupo
    request.allowedLocalIds = todosLosLocales.map(l => l.id)
  }
}

Todos los queries de cajas aplican el filtro:

where: { id_local: { in: request.allowedLocalIds } }
El header X-App-Id es obligatorio en todas las rutas de cajas. El frontend lo inyecta automáticamente en cada request Axios a través del interceptor de api/client.js leyendo activeApp.app.id del store persistido.

Flujos de operación

Ciclo de vida de una caja

1 · Apertura de turno

El cajero hace click en "+ Nueva Caja", completa fecha_inicio y opcionalmente nro_turno y cajero. El backend crea la caja con fecha_cierre: null, lo que indica turno abierto.

2 · Registro durante el turno

Desde el DrawerPanel de detalle, el operador agrega Movimientos (INGRESO/EGRESO/APERTURA por método) y Detalles (Total Digitales, MP QR, etc.).

3 · Cierre del turno

El cajero edita la caja (PUT /:id) completando fecha_cierre, total, efectivo, fiscal, comensales y tickets. Los totales son manuales — el sistema no los calcula desde movimientos.

4 · Consulta posterior

La caja queda visible en la lista y puede abrirse para ver el historial completo de movimientos y detalles. Los stats del Dashboard agregan los campos total y efectivo de todas las cajas del período.
