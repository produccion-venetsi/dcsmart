## 1. Resumen ejecutivo

DCSmart es un backoffice web multi-tenant para grupos gastronómicos. Permite gestionar múltiples marcas (apps) cada una con sus propios locales físicos, cajas, pagos a proveedores y usuarios con diferentes niveles de acceso. El sistema implementa aislamiento estricto de datos por app y por local mediante un modelo RBAC combinado con control de acceso basado en atributos.

### Modelo multi-tenant

```
App (grupo gastronómico)
 ├── Local A
 │     ├── Cajas
 │     │    ├── Movimientos
 │     │    └── Detalles
 │     └── Pagos
 ├── Local B
 │     └── ...
 └── Usuarios
       ├── super_admin  →  todos los locales
       ├── admin        →  todos o locales específicos (UserLocalAccess)
       └── cajero       →  locales específicos (UserLocalAccess)
```

Cada request autenticado que accede a datos operativos debe incluir el header `X-App-Id`. El middleware `appContext` resuelve qué locales puede ver el usuario dentro de esa app y lo expone como `request.allowedLocalIds[]`. Todos los queries filtran por ese array.

## 3. Autenticación y autorización

### 3.1 Autenticación

**JWT:** Todos los endpoints protegidos requieren `Authorization: Bearer <token>`. El token también se almacena en cookie `httpOnly` (para PWA). Expiración configurable vía `JWT_EXPIRES_IN` (default: `7d`).

**Flujos de login:**

```
Email/Password
  POST /api/auth/login  { email, password }
    → bcryptjs.compare(password, hash)
    → JWT firmado con user.id + email
    → Respuesta: { token, user }

Google OAuth
  POST /api/auth/google  { credential }
    → google-auth-library verifyIdToken(credential)
    → Busca user por google_id o email
    → Si email existe sin google_id: asocia automáticamente
    → Si no existe: crea user nuevo
    → JWT firmado
    → Respuesta: { token, user }
```

**Sesión en frontend:** El token se persiste en `localStorage` vía `authStore` (Zustand). El interceptor Axios lo inyecta en cada request.

### 3.2 Autorización (RBAC)

**Roles base:**

| Rol | Descripción |
|-----|-------------|
| `super_admin` | Acceso total a todas las apps y todos los locales |
| `admin` | Acceso completo a la app asignada (view/create/edit en todos los módulos) |
| `cajero` | Acceso view/create solo en módulos de caja |

**Módulos del sistema:**

`caja`, `caja_movimientos`, `pagos`, `proveedores`, `rubros`, `categorias`, `usuarios`, `apps`, `locales`

**Permisos:**

```
RolePermission: { id_role, id_module, can_view, can_create, can_edit, can_delete }
UserPermission: { id_user, id_module, ... }  ← override individual de rol
```

El decorator `fastify.can(módulo, acción)` primero busca `UserPermission` del usuario; si no existe, usa `RolePermission` del rol asignado. Responde 403 si no tiene permiso.

### 3.3 Aislamiento por app y local (AppContext)

**Header requerido:** `X-App-Id: <uuid>`

**Flujo del middleware `appContext`:**

```
1. Lee X-App-Id del header → error 400 si falta
2. Busca UserAppRole(id_user, id_app) → error 403 si no tiene acceso
3. Si rol == 'super_admin':
     allowedLocalIds = todos los locales activos de la app
4. Si rol != 'super_admin':
     a. Busca UserLocalAccess(id_user, id_app)
     b. Si hay registros: allowedLocalIds = esos locales
     c. Si no hay registros: allowedLocalIds = todos los locales de la app
5. Setea en request: activeAppId, isSuperAdmin, allowedLocalIds[]
```

**Uso en queries:**
```javascript
// Todos los filtros operativos usan este patrón:
const where = {
  id_local: { in: request.allowedLocalIds },
  ...otrosFiltros
}
```

**Tabla `user_local_access`:** Permite restringir a un cajero para que solo vea los datos de su local específico dentro de un grupo.

```
UserLocalAccess { id_user, id_app, id_local }
@@unique([id_user, id_app, id_local])
```

### 6.2 Plugins registrados en server.js

```
@fastify/cors     origin: FRONTEND_URL, credentials: true
@fastify/jwt      secret: JWT_SECRET, cookie: 'token'
@fastify/cookie   httpOnly
db                PrismaClient → fastify.db
permissions       → fastify.can()
appContext        → fastify.appContext
```

**Decorator global `fastify.authenticate`:** valida JWT en header Authorization Bearer.

### 6.3 Convenciones de código

- **ESModules** (`import/export`) en todo el proyecto
- **`async/await`** — nunca callbacks
- **UUIDs v4** para todos los IDs (`@default(uuid())`)
- **Timestamps UTC** (`@default(now())`, `@updatedAt`)
- **Decimales monetarios** `Decimal(12,2)`
- **Nombres de tablas** en plural y snake_case (`@@map`)
- **Nombres de campos** en snake_case
- **Rutas protegidas por defecto** — las públicas son la excepción
- **Soft-delete** para usuarios y proveedores (`activo: false`)
- **Errores HTTP correctos:** 400 validación, 401 no auth, 403 sin permiso, 404 no encontrado, 500 error interno

### 7.2 Zustand Stores

**`authStore.js`** — Persistencia: `dcsmart-auth` (localStorage)
```
State:  user, token, loading, error
Actions:
  login(email, password)       POST /api/auth/login
  loginGoogle(credential)      POST /api/auth/google
  logout()                     Limpia token + clearContext()
  refreshUser()                GET /api/auth/me
  clearError()
```

**`appStore.js`** — Persistencia: `dcsmart-app-context` (localStorage)
```
State:  activeApp, activeLocal
  activeApp shape: { app: { id, nombre, slug }, role, locales: [{ id, nombre }] }
  activeLocal shape: { id, nombre } | null
Actions:
  setActiveApp(app)    Setea app activa, limpia local
  setActiveLocal(l)    Setea local activo
  clearContext()       Limpia ambos
```

**`uiStore.js`** — Sin persistencia
```
State:  sidebarOpen (default: true), notifications[]
Actions:
  toggleSidebar()
  setSidebarOpen(open)
  notify(message, type)        Auto-dismiss en 4 segundos
  addNotification(n)
  removeNotification(id)
```

### 7.3 API Client (Axios)

**`src/api/client.js`:**
```
BaseURL: /api
Interceptors:
  REQUEST:
    - Lee token de authStore → Authorization: Bearer <token>
    - Lee activeApp de appStore → X-App-Id: <activeApp.app.id>
  RESPONSE:
    - 401 → limpia token, redirige a /login
    - 403 → log de error
```

### 7.4 Routing

```
/login                       Login.jsx              (público)
/select-app                  AppSelector.jsx        (JWT requerido)
/dashboard                   Dashboard.jsx          (JWT + app)
/cajas                       CajaList.jsx           (JWT + app)
/cajas/:id                   CajaDetail.jsx         (JWT + app)
/pagos                       PagoList.jsx           (JWT + app)
/pagos/nuevo                 PagoForm.jsx           (JWT + app)
/pagos/:id/editar            PagoForm.jsx           (JWT + app)
/proveedores                 ProveedorList.jsx      (JWT + app)
/proveedores/nuevo           ProveedorForm.jsx      (JWT + app)
/proveedores/:id/editar      ProveedorForm.jsx      (JWT + app)
/admin/users                 Users.jsx              (JWT + app)
/admin/apps                  Apps.jsx               (JWT + app)
/admin/locales               Locales.jsx            (JWT + app)
/admin/rubcat                RubCat.jsx             (JWT + app)
/admin/metodos-pago          MetodosPago.jsx        (JWT + app)
/admin/roles                 Roles.jsx              (JWT + app)
```

Todas las páginas usan **lazy loading** (`React.lazy` + `Suspense`). `ProtectedRoute` con `requireApp={true}` redirige a `/select-app` si no hay app activa.

### 7.6 Componentes reutilizables

**`DrawerPanel`** — Modal lateral vía React Portal
- Ancho configurable (default 540px)
- Cierra con click en backdrop o tecla ESC
- Bloquea scroll del body cuando está abierto

**`Sidebar`** — Navegación principal
- Selector de app activa y local (dropdown)
- Links de navegación con iconos SVG inline
- Sección admin (solo visible si tiene permisos)
- Footer con avatar, nombre y logout

**`Layout`** — Wrapper de rutas autenticadas
- Renderiza Sidebar + `<Outlet>`
- Renderiza `NotificationsContainer` (toasts auto-dismiss 4s)

**`ProtectedRoute`** — Guard de autenticación
- `requireApp={false}`: solo verifica token JWT
- `requireApp={true}`: verifica token + `activeApp` en store

**`AppLogo`** — Logo oficial
- Props: `variant` ("horizontal"), `style`
- Carga imagen desde `/logos/DCSMART-APP-{variant}.png`

## 8. CI/CD y despliegue

### 8.1 GitHub Actions — `.github/workflows/deploy.yml`

```yaml
Trigger: push a rama "master"
Runner:  ubuntu-latest
Node:    22

Pasos:
  1. actions/checkout@v4
  2. actions/setup-node@v4  (node 22, cache npm)
  3. npm ci  (en frontend/)
  4. npm run build  (genera frontend/dist/)
  5. firebase deploy --only hosting --project dc-smart-mvp
     (secret: FIRABASETOKEN)
```

**Nota:** El workflow solo despliega el frontend. El backend en Cloud Run se despliega manualmente o por un proceso separado.

### 8.2 Firebase Hosting — `firebase.json`

```json
{
  "hosting": {
    "site": "dc-smart-mvp",
    "public": "frontend/dist",
    "rewrites": [
      { "source": "/api/**",
        "run": { "serviceId": "dcsmart-backend", "region": "us-central1" } },
      { "source": "**", "destination": "/index.html" }
    ]
  }
}
```

- URL pública: `https://dc-smart-mvp.web.app`
- `/api/**` → Cloud Run `dcsmart-backend` (us-central1)
- Todo lo demás → `index.html` (SPA routing)

## 11. Estado actual del proyecto

### Implementado ✅

- Schema de base de datos completo (21 modelos, 5 enums)
- Autenticación: email/password + Google OAuth + JWT
- RBAC completo: roles, módulos, permisos por rol y por usuario
- Aislamiento multi-tenant por app y local (header `X-App-Id`)
- 13 módulos API REST con CRUD completo
- Frontend SPA: 18 páginas, Zustand stores, interceptores Axios
- PWA: vite-plugin-pwa, manifest, service worker
- CI/CD: GitHub Actions → Firebase Hosting en push a master
- Despliegue: Firebase Hosting + Cloud Run + Cloud SQL

### Pendiente / Deuda técnica ⚠️

- Google OAuth: `VITE_GOOGLE_CLIENT_ID` y `GOOGLE_CLIENT_ID` no configurados en producción
- Sin validación global de inputs en frontend (no hay Zod ni similar)
- Sin tests unitarios ni e2e
- Sin paginación cursor-based (usa offset, puede ser ineficiente a escala)
- Sin manejo de errores global en frontend (manejo ad-hoc por componente)
- No responsive para mobile (diseñado para desktop/tablet)
- Sin upload de imágenes implementado (`foto_url`, `pdf_url` son strings de URL manual)
- `GOOGLE_CLIENT_ID` y `GOOGLE_CLIENT_SECRET` vacíos — login con Google inhabilitado
